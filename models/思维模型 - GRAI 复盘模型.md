# GRAI 复盘模型 (GRAI Review Model)

## 概述

中国本土化的复盘方法论，源自军队 AAR（After Action Review）和联想等企业的实践。**GRAI = Goal（目标）→ Result（结果）→ Analysis（分析）→ Insight（洞察）**。核心精神：**复盘不是追究责任，而是从经验中学习。**

## 四步法

| 步骤 | 核心问题 | 关键动作 |
|------|---------|---------|
| **G - Goal 目标** | 原始目标是什么？ | 回顾最初的目的，不要事后美化 |
| **R - Result 结果** | 实际结果是什么？ | 诚实呈现数据和事实，不回避失败 |
| **A - Analysis 分析** | 差异产生的原因？ | 成功的关键因素 + 失败的根本原因 |
| **I - Insight 洞察** | 学到了什么？ | 可复用的经验 + 下次怎么做得更好 |

## GRAI vs 传统总结

| | 传统工作总结 | GRAI 复盘 |
|--|------------|-----------|
| 目的 | 交代/汇报/表功 | 学习与改进 |
| 基调 | "我做了很多" | "发生了什么？为什么？" |
| 对待失败 | 避讳、淡化、找借口 | **当作最宝贵的学习素材** |
| 输出 | 一份漂亮的报告 | 行动规则和流程改进 |
| 责任导向 | 追究"谁的错" | 关注"哪里可以改进" |

## 使用步骤

1. **回顾 Goal** — 当初为什么做这件事？目标是什么？
2. **对比 Result** — 做得怎么样？亮点有哪些？不足有哪些？（用数据）
3. **深度 Analysis**：
   - 成功：是实力还是运气？能复制吗？
   - 失败：根本原因是什么？（用 5 Whys 挖掘）
4. **提炼 Insight**：
   - **Continue Doing** — 继续做什么
   - **Start Doing** — 开始做什么
   - **Stop Doing** — 停止做什么
5. **形成行动计划** — 下一次什么时候用上这些洞察？

## 示例

> **一个项目延期的 GRAI 复盘：**
>
> - **G**：原定 6 周上线 MVP，覆盖核心 3 个功能
> - **R**：实际用了 10 周，只上线了 2 个功能，Bug 率 15%
> - **A**：
   - ✅ 成功原因：技术方案选型好，团队学习能力强
   - ❌ 失败原因（5 Whys）：延期 → 需求变更频繁 → 需求没锁死 → 产品经理没有变更管理流程 → **根因：缺少需求冻结机制**
> - **I**：
   - Continue：技术选型的评估流程
   - Start：需求冻结制度 + 变更审批流程
   - Stop：口头确认需求后直接开发

## 注意事项

- **复盘要趁热**——隔太久记忆失真，最好在项目结束后一周内
- **对事不对人**——如果变成批斗会，没人敢说真话
- **不仅要复盘失败，也要复盘成功**——成功中的运气部分下次可能不在
- 洞察必须**可执行**——"下次注意"不是洞察，"下次增加代码评审环节"才是

## 变体/相关模型

- **KISS 复盘模型**：更轻量的版本（Keep, Improve, Start, Stop）
- **AAR (After Action Review)**：美军版本，GRAI 的源头之一
- **5 Whys**：Analysis 阶段的根因分析工具
- **PDCA 循环**：复盘后的持续改进大框架
- ** retrospectives（回顾会议）**：敏捷开发中的复盘实践

## 适用场景

- 项目结束后复盘
- 季度/年度复盘
- 重大事件后反思（危机、发布、活动）
- 个人定期自我复盘（周/月/季）
- 团队迭代改进
- 失败案例学习
