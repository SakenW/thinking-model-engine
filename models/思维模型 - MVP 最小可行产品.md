# MVP 最小可行产品 (Minimum Viable Product)

## 概述

Eric Ries 在《精益创业》中提出：用**最少的资源、最快的时间**构建一个能验证核心假设的产品版本，投入市场获取真实反馈，然后迭代。MVP 不是"半成品"，而是"刚好够验证假设的成品"。

## MVP 的核心思维

```
传统做法：                    MVP 做法：
  
  想法 → 开发 → 发布          想法 → 假设 → 最小实验 → 验证/证伪 → 迭代
       (6-12个月)                   (1-4周)
       
       全功能产品                      可能只是一个落地页
       
    "我相信用户需要"              "让我证明用户真的需要"
```

## 常见 MVP 类型

| 类型 | 说明 | 适用场景 | 成本 |
|------|------|---------|------|
| **落地页 (Landing Page)** | 一个页面描述产品概念，收集邮箱 | 验证需求是否存在 | 极低 |
| **Wizard of Oz（绿野仙踪）** | 前端看起来像自动化，后端人工操作 | 验证用户是否愿意付费使用 | 低 |
| ** Concierge（门房服务）** | 完全人工为少量用户提供服务 | 验证解决方案是否有效 | 中 |
| **单功能版** | 只做最核心的一个功能 | 验证核心价值主张 | 中 |
| **众筹页面** | 在 Kickstarter 等平台展示概念 | 验证付费意愿 + 融资 | 低 |

## 使用步骤

1. **明确要验证的核心假设** — "我认为______"（一个假设）
2. **定义成功标准** — 什么数据证明假设成立？（注册率？付费率？留存率？）
3. **选择 MVP 形式** — 用最小成本验证
4. **设定时间盒** — 1-2 周内必须拿到结果
5. **执行 & 收集反馈**
6. **决策：转型 (Pivot) 还是坚持 (Persevere)**

## 示例

> **Dropbox 的 MVP：**
>
> - 没有写一行代码
> - 只做了一个 3 分钟演示视频，展示"文件同步"的概念
> - 放到 Hacker News → 一夜之间从 5000 等待列表跳到 75000
> - **结论：需求真实存在** → 开始开发
>
> **Zappos 的 MVP（绿野仙踪）：**
>
> - 没有库存，没有仓库
> - 把本地鞋店的照片放到网上
> - 有人下单 → 去鞋店买 → 寄给客户
> - 验证了"人们愿意在网上买鞋"

## 注意事项

- **MVP ≠ 偷工减料**——用户体验不能差到无法判断
- **MVP ≠ 半成品**——它必须是可用的，只是范围极小
- 不要把 MVP 当成不认真做的借口
- 内部工具/企业软件的 MVP 思路不同——**客户期望更高**

## 变体/相关模型

- **精益创业循环 (Build-Measure-Learn)**：MVP 所在的大框架
- **设计冲刺 (Design Sprint)**：Google Ventures 的 5 天 MVP 方法论
- **RICE 优先级模型**：决定先做哪个功能的优先级工具
- **第一性原理**：找到"最小可行"中的那个核心

## 适用场景

- 创业公司产品验证
- 新功能上线前的测试
- 新市场进入探索
- 商业模式验证
- 个人项目启动（不要憋大招）
