# 思维模型引擎 (Thinking Model Engine)

基于 117 个思维模型的**问题处方系统**。不做"图书馆检索员"，做"全科医生"——诊断问题 → 开处方 → 给行动方案。

## 模型库位置

技能目录下的 `models/` 子目录包含全部 117 个思维模型 .md 文件。
使用时通过相对路径读取：`models/思维模型 - [文件名].md`

> 注：这些文件从 Obsidian 知识库复制而来，是发布包的一部分。本地原始位置：
> `/Users/saken/Documents/Obsidian/Saken/30_参考资料/思维模型/`

## 核心工作流

### STEP 1: 接收与判断

1. **理解用户问题** — 提取核心诉求和背景信息
2. **判断是否适合处理** — 参考 `scenario-prescriptions.md` 底部的排除清单：
   - ❌ 纯查询 / 纯执行 / 纯情绪宣泄 / 极度紧急 / 已指定单一模型 → **不使用本技能**
   - ✅ 有明确问题需要分析 → 继续

### STEP 2: 问题分类与匹配

1. **加载 `references/scenario-prescriptions.md`**
2. **匹配问题类型** — 将用户问题归入 P0-P9 之一：
   - P0 🔥 决策困境
   - P1 🎯 目标达成困难
   - P2 📚 学习新领域
   - P3 ⏰ 效率低下
   - P4 😤 情绪/人际
   - P5 💼 工作/团队/项目
   - P6 🧠 战略/职业
   - P7 💡 创意/突破
   - P8 🔍 复盘/总结
   - P9 🎪 说服/谈判
3. **选择版本** — 轻量版还是深度版：
   - 用户想要快速答案 / 问题相对简单 → **轻量版**（2-3 个模型）
   - 用户说"深入分析""帮我好好想想" / 问题复杂重大 → **深度版**（5-8 个模型）
4. **确定模型列表** — 从处方中选出本次要用的模型，标注每个的角色：
   - `主分析`：核心框架
   - `防偏误`：防止认知偏差
   - `补充`：额外视角

### STEP 3: 读取模型原文

1. **加载 `references/model-quickref.md`** 获取文件名映射
2. **只读取选中的 2-5 个模型 .md 文件**（按需加载，不全量读入）
3. **提取重点**：核心机制 + 使用步骤 + 模板 + 关键示例

> 文件路径格式：`models/思维模型 - [文件名].md`

### STEP 4: 组合分析与输出

按照以下固定格式输出：

```markdown
## 🔍 诊断

[用 2-3 句话概括问题的本质是什么]

## 🧠 分析框架

[列出使用的每个模型 + 一句话说明为什么用它]

## 📊 分析过程

[用选中的模型框架逐步展开分析。每个模型的输出要有实质内容，
不是简单罗列模型名。模型之间要有交叉引用和递进关系。]

## ✅ 行动方案

[具体的、可执行的、有时间线的行动步骤。
不是"建议你…"而是"明天你可以…"]

## ⚠️ 陷阱提醒

[基于防偏误模型，指出这个决策/行动中常见的认知陷阱]

## 🔗 深挖方向

[如果用户想进一步深入，还有哪些模型可以补充分析。
**这是必须有的结尾段落——每次输出都要提示用户是否需要深挖。**]
```

### 输出原则

1. **处方 > 索引** — 给出分析和方案，不是列一堆模型让用户自己选
2. **中文优先** — 用中文场景和例子，避免翻译腔
3. **具体可操作** — 行动方案要精确到"明天做什么"，不要泛泛而谈
4. **模型有角色** — 说明每个模型在这次分析中扮演什么角色
5. **承认局限** — 低可信度（⭐）的模型要标注"这是启发性的参考"
6. **深挖提示必带** — 每次输出的最后一段必须是深挖邀请

### STEP 5: 可选深挖（用户触发）

当用户回应"深挖一下""再详细说说""继续分析"时：

1. **加载 `references/combo-patterns.md`** 找到对应配方
2. **追加更多模型**（通常再加 3-5 个）
3. **在前次分析基础上深化**，不是从头开始
4. **输出更详细的版本**，可以包含：
   - 完整的模板填写示例
   - 更多的二阶效应分析
   - 与其他相关案例的对比
   - 长期跟踪建议

### STEP 6: 后处理（可选，视上下文决定）

分析完成后，根据用户需求可建议对接：

| 如果用户想… | 建议 |
|-------------|------|
| 把结论变成任务 | 对接 taskflow 或 GTD 系统 |
| 记录到知识库 | 写入 Obsidian / Notion |
| 做定期回顾 | 设定 reminder |
| 分享给他人 | 用金字塔原理/PREP 结构化表达 |

## 模型可信度分级

使用模型时注意区分可信度：

| 分级 | 含义 | 使用方式 |
|------|------|---------|
| ⭐⭐⭐ | 经典理论，有坚实学术/实践基础 | **首选**，可作为主要分析框架 |
| ⭐⭐ | 业界广泛验证，但非严格学术或存在争议 | **常用**，标注适用边界 |
| ⭐ | 自创/小众/启发性工具 | **辅助**，作为补充视角，不单独依赖 |
